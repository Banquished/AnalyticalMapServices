<objective>
Replace the silent `return []` fallback in the three ArcGIS REST clients with a typed `ApiError`
raise so that API-level errors are distinguishable from genuine empty results.

Purpose: Currently an ArcGIS API error (e.g., invalid query, rate limit, service outage) produces
the same result as "no features at this location" — silently empty. This masks real failures and
makes debugging impossible. Raising `ApiError` ensures errors are logged with context and visible
in the call stack.

Output: Three patched client files where `"error" in data` raises `ApiError` instead of logging
and returning `[]`. The `fetch_json` `data is None` path (network failure, already logged) still
returns `[]` as before.
</objective>

`ApiError` already exists in `backend/src/app/clients/http.py`:

```python
class ApiError(Exception):
    kind: str
    def __init__(self, kind: str, message: str, status: int | None = None) -> None:
        super().__init__(message)
        self.kind = kind
        self.status = status
```

How `feature_info.py` consumes these clients (confirms safe raise):

```python
all_results = await asyncio.gather(*wms_tasks, *rest_tasks, return_exceptions=True)

# _rest() already handles BaseException — ApiError is a subclass
def _rest(idx: int) -> list[dict]:
    result = all_results[n_wms + idx]
    if isinstance(result, BaseException):
        logger.warning("REST query %d failed: %s", idx, result)
        return []          # graceful degradation — no crash
    return cast(list[dict], result)
```
